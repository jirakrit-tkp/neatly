# ฟีเจอร์: ระบบยืนยันตัวตนและการอนุญาต

## 1. ภาพรวม

ระบบยืนยันตัวตนจัดเตรียมการลงทะเบียนผู้ใช้ การเข้าสู่ระบบ การจัดการเซสชัน และการควบคุมการเข้าถึงตามบทบาทอย่างปลอดภัยสำหรับแพลตฟอร์ม Neatly Hotel รองรับทั้งผู้ใช้ที่ยืนยันตัวตนแล้ว (ลูกค้าและแอดมิน) และผู้ใช้แบบไม่ระบุตัวตน (สำหรับฟังก์ชันแชทบอท) ระบบสร้างบน Supabase Auth พร้อมการจัดการโปรไฟล์แบบกำหนดเองและการเชื่อมโยงเซสชันอัตโนมัติเมื่อผู้ใช้แบบไม่ระบุตัวตนเข้าสู่ระบบ

**วัตถุประสงค์:**
- การยืนยันตัวตนผู้ใช้และการจัดการเซสชันอย่างปลอดภัย
- การควบคุมการเข้าถึงตามบทบาท (ลูกค้า, แอดมิน)
- รองรับผู้ใช้แบบไม่ระบุตัวตน/Guest สำหรับแชทบอท
- การสร้างและจัดการโปรไฟล์
- การยืนยันอีเมล
- ฟังก์ชันรีเซ็ตรหัสผ่าน
- การเชื่อมโยงเซสชันจาก guest เป็น user อัตโนมัติ

**จุดเชื่อมต่อ:**
- เส้นทางที่มีการป้องกันทั้งหมด (แผงแอดมิน, การจอง, โปรไฟล์)
- ระบบแชทบอท (แบบไม่ระบุตัวตน + ยืนยันตัวตนแล้ว)
- ระบบจอง (ต้องยืนยันตัวตน)
- การจัดการโปรไฟล์

## 2. สถาปัตยกรรม / ลำดับการทำงาน

### ลำดับการลงทะเบียนผู้ใช้
```
ผู้ใช้กรอกฟอร์มลงทะเบียน → การตรวจสอบฝั่ง Frontend
  → AuthService.registerUser() → Supabase Auth (สร้างผู้ใช้)
  → สร้างโปรไฟล์ในตาราง `profiles` → อัปโหลดรูปโปรไฟล์ (ถ้ามี)
  → ส่งอีเมลยืนยันตัวตน → เปลี่ยนเส้นทางไปหน้าเข้าสู่ระบบ
```

### ลำดับการเข้าสู่ระบบ
```
ผู้ใช้ป้อนข้อมูลรับรอง → Supabase Auth.signInWithPassword()
  → สร้างเซสชัน → AuthContext อัปเดต → ตรวจสอบ anonymous_id ใน localStorage
  → เชื่อมโยงเซสชันแชทบอทของ guest (ถ้ามี) → เปลี่ยนเส้นทางตามบทบาท
```

### ลำดับผู้ใช้แบบไม่ระบุตัวตน (Guest)
```
ผู้ใช้เปิดเว็บไซต์โดยไม่เข้าสู่ระบบ → AuthContext สร้าง anonymous_id
  → เก็บใน localStorage → ใช้สำหรับเซสชันแชทบอท
  → เมื่อเข้าสู่ระบบ: เชื่อมโยงเซสชัน guest กับบัญชีผู้ใช้ → ล้าง anonymous_id
```

### ลำดับการเชื่อมโยงเซสชัน
```
ตรวจพบเหตุการณ์ SIGNED_IN → รับ anonymous_id จาก localStorage
  → เรียก /api/chat/link-session → อัปเดตตาราง chatbot_sessions
  → ตั้งค่า customer_id สำหรับเซสชันแบบไม่ระบุตัวตนทั้งหมด → ล้าง anonymous_id
```

## 3. เทคโนโลยีและไลบรารี

| ไลบรารี/API | วัตถุประสงค์ | ตัวอย่างการใช้งาน |
|------------|---------|---------------|
| `@supabase/supabase-js` | Backend สำหรับยืนยันตัวตน | การลงทะเบียน, เข้าสู่ระบบ, การจัดการเซสชัน |
| Supabase Auth | การจัดการข้อมูลระบุตัวตน | OAuth, JWT tokens, การเก็บเซสชัน |
| Supabase Storage | รูปโปรไฟล์ | อัปโหลด/ดึงรูปอวาตาร์ผู้ใช้ |
| React Context API | สถานะยืนยันตัวตนส่วนกลาง | `AuthContext`, `AuthProvider` |
| localStorage | เก็บ Anonymous ID | ติดตามเซสชัน Guest |
| Next.js middleware | การป้องกันเส้นทาง | การป้องกันเส้นทาง Admin/ลูกค้า |

## 4. ตรรกะหลัก

### AuthService (src/services/authService.ts)

**การลงทะเบียนผู้ใช้:**
```typescript
1. สร้างผู้ใช้ใน Supabase Auth (auth.users)
2. แทรกโปรไฟล์ในตาราง `profiles` พร้อมข้อมูลผู้ใช้
3. อัปโหลดรูปโปรไฟล์ไปยัง Supabase Storage (ถ้ามี)
4. อัปเดตโปรไฟล์ด้วย URL รูปภาพ
5. ส่งอีเมลยืนยันตัวตน
```

**การตรวจสอบชื่อผู้ใช้ว่าง:**
```typescript
Query ตาราง profiles เพื่อหาชื่อผู้ใช้ที่มีอยู่
Return: true ถ้าว่าง, false ถ้ามีคนใช้แล้ว
```

### AuthContext (src/context/AuthContext.tsx)

**การจัดการสถานะ:**
- `user`: ผู้ใช้ที่ยืนยันตัวตนในปัจจุบัน (User | null)
- `loading`: ธงสถานะการโหลดการยืนยันตัวตน
- `anonymousId`: UUID ที่สร้างขึ้นสำหรับผู้ใช้แบบ guest
- `isGuest`: ธงคำนวณ (ไม่มี user + มี anonymousId)

**ตรรกะการเชื่อมโยงเซสชัน:**
```typescript
เมื่อเกิดเหตุการณ์ SIGNED_IN:
1. รับ anonymous_id จาก localStorage
2. เรียก /api/chat/link-session ด้วย anonymousId
3. Backend อัปเดต chatbot_sessions: ตั้งค่า customer_id
4. ล้าง localStorage anonymous_id
5. ผู้ใช้เห็นประวัติแชทของ guest
```

**การสร้าง Anonymous ID:**
```typescript
function getOrCreateAnonymousId():
  ตรวจสอบ localStorage หา 'neatly_anonymous_id'
  ถ้าไม่มี:
    สร้าง UUID v4
    เก็บใน localStorage
  Return anonymous_id
```

```sql
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id),
  username TEXT UNIQUE NOT NULL,
  first_name TEXT,

  country TEXT,
  profile_image TEXT,
```

## 6. กรณีพิเศษ / ข้อจำกัด / TODO

### กรณีพิเศษที่จัดการแล้ว
1. **ชื่อผู้ใช้ซ้ำ**: ตรวจสอบก่อนลงทะเบียนผ่าน `checkUsernameAvailability()`
2. **การยืนยันอีเมล**: ผู้ใช้สามารถส่งอีเมลยืนยันอีกครั้งผ่าน `resendVerificationEmail()`
3. **การเชื่อมโยงเซสชัน Guest**: ป้องกันเซสชันซ้ำเมื่อผู้ใช้เข้าสู่ระบบหลายครั้ง
4. **การแทนที่รูปโปรไฟล์**: ลบรูปเก่าก่อนอัปโหลดใหม่
5. **การเก็บเซสชัน**: เปิดใช้งานการรีเฟรช token อัตโนมัติใน Supabase client

### ข้อจำกัดปัจจุบัน
1. **ไม่มี Social OAuth**: มีเฉพาะการยืนยันตัวตนด้วยอีเมล/รหัสผ่าน (ไม่มี Google, Facebook ฯลฯ)
2. **ไม่มีการยืนยันตัวตนสองขั้นตอน (2FA)**: เข้าสู่ระบบด้วยรหัสผ่านอย่างเดียว
3. **ไม่สามารถลบบัญชี**: ผู้ใช้ไม่สามารถลบบัญชีด้วยตนเองได้
4. **ไม่สามารถเปลี่ยนบทบาท**: ไม่สามารถเปลี่ยนบทบาทผู้ใช้ (ลูกค้า ↔ แอดมิน) ผ่าน UI
5. **ข้อจำกัดแชทบอท Guest**: ผู้ใช้แบบไม่ระบุตัวตนจะสูญเสียประวัติแชทถ้าล้าง localStorage

### ปัญหาที่ทราบ
- **การเก็บ Anonymous ID**: ถ้าผู้ใช้ล้างข้อมูลเบราว์เซอร์ เซสชันแชทบอทของ guest จะหายไป
